# 基于 sub2api 专属分组的兑换权限设计

## 背景

余额兑换码工具当前允许所有通过 sub2api 用户 Token 完成身份验证的用户使用。现在需要把兑换能力限制为指定专属分组成员，且不修改 sub2api 本身。

管理员已在 sub2api 创建并启用专属分组“分销代理”，分组 ID 为 `24`。本工具继续使用 sub2api 用户资料接口作为身份和权限的唯一可信来源。

## 目标

- 只有 `allowed_groups` 包含分组 ID `24` 的用户可以访问和执行兑换。
- 用户被移出该分组后，下一次受保护请求立即失去权限。
- 未授权用户看到明确的无权限页面，而不是“会话已失效”。
- 权限校验无法通过隐藏前端、伪造用户 ID、伪造分组 ID 或直接调用接口绕过。
- 保持现有批量兑换、限流、幂等、余额刷新和未完成操作恢复行为不变。

## 非目标

- 不修改 sub2api 源码、数据库或自定义菜单能力。
- 不在本工具内建设用户或分组管理后台。
- 不支持多个允许分组、按用户白名单或管理员角色自动放行。
- 不保证 sub2api 菜单对未授权用户隐藏；sub2api 当前只能按普通用户或管理员控制菜单可见性。

## 配置

新增必填环境变量：

```env
REDEEM_ALLOWED_GROUP_ID=24
```

该配置只接受正整数。缺失、填写 `#24`、零、负数、小数或其他非数字内容时，服务必须在启动阶段失败，避免误配置导致全员可用。

配置类型 `AppConfig` 增加允许兑换的分组 ID，并以数字形式传入身份校验逻辑。`.env.example` 和 README 部署教程同步说明此变量的用途和填写方法。

## 上游资料模型

本地 sub2api profile schema 增加 `allowed_groups` 字段。合法值为正整数数组；该字段缺失或格式非法时规范化为空数组，以便安全地判定为未授权。授权只信任 `GET /api/v1/user/profile` 的最新响应，不读取浏览器提供的用户 ID 或分组 ID。

以下情况均按未授权处理：

- `allowed_groups` 不包含配置的分组 ID；
- `allowed_groups` 缺失；
- `allowed_groups` 不是数组或包含非法值；

除 `allowed_groups` 之外的必需 profile 字段不能通过本地 schema 校验，以及上游不可访问、超时或返回无法解析的异常响应，仍沿用现有“上游服务不可用”错误，不伪装成无权限。

## 服务端授权流程

在现有 session 身份读取与 profile 重验证边界中加入统一授权检查。检查条件为：

```text
profile.allowed_groups.includes(config.redeemAllowedGroupId)
```

校验覆盖以下入口：

1. `POST /api/session/exchange`：验证 Token、读取最新 profile，并在创建会话 Cookie 前检查分组权限。首次登录无权限时不创建新 Cookie。
2. `GET /api/me`：重新读取最新 profile 并检查权限。
3. `POST /api/conversions/prepare`：在创建操作令牌前重新检查权限。
4. `POST /api/conversions/execute`：在生成兑换码或扣减余额前重新检查权限。
5. 未完成操作恢复：恢复过程复用 prepare 或 execute，因此同样经过最新权限检查。

每个请求内仍复用 `SessionReader` 的请求级缓存，避免同一请求重复请求上游；不同请求之间不缓存授权结果。这样用户被移出分组后，下一次请求立即失效，重新加入后下一次重验证立即恢复。

授权失败抛出统一的 `AppError`：

```text
HTTP 状态：403
错误码：REDEEM_ACCESS_DENIED
安全消息：暂无余额兑换权限，请联系管理员
```

授权失败不清除已有会话 Cookie，因为用户身份仍然有效，仅兑换权限不足。身份 Token 无效或过期时仍返回现有 `401` 会话错误并清除 Cookie。

## 首次登录与重新检查

首次使用 URL Token 打开工具但尚未加入分组时：

- Token 验证成功，但 exchange 返回 `403`；
- 服务端不创建新会话 Cookie；
- 前端在当前页面内暂存已经读取的 Token，以支持“重新检查”；
- 管理员将用户加入分组后，用户点击“重新检查”会再次执行 exchange；
- 如果用户刷新整个页面，内存中的 Token 会消失，需要返回 sub2api 重新打开工具。

已有会话的用户被移出分组时：

- `/api/me`、prepare 或 execute 返回 `403`；
- 前端进入无权限状态但保留会话 Cookie；
- 重新加入分组后点击“重新检查”，通过 `/api/me` 恢复正常状态。

## 前端状态与界面

前端会话状态增加独立的 `unauthorized` 状态，并把 `REDEEM_ACCESS_DENIED` 映射到该状态。

无权限页面显示：

- 标题：`暂无余额兑换权限`
- 说明：`当前账号未加入“分销代理”专属分组，请联系管理员。`
- 操作：`重新检查`

处于无权限状态时不显示余额、兑换表单、兑换结果、历史记录、未完成操作恢复按钮和确认弹窗。该页面复用现有会话状态区域的布局和图标风格，不新增独立页面或路由。

如果 prepare、execute 或恢复操作期间权限被撤销，前端立即清除当前展示的 profile 并进入无权限状态。浏览器本地保存的未完成操作数据不删除，避免在重新授权后丢失恢复线索；但无权限期间不展示也不能执行这些操作。

## 错误分类

- `REDEEM_ACCESS_DENIED`：HTTP `403`，不可重试型业务错误，进入无权限状态。
- `SESSION_REQUIRED`、`SESSION_INVALID`、`SESSION_EXPIRED`：HTTP `401`，进入会话失效状态。
- `UPSTREAM_UNAVAILABLE`：HTTP `502`，进入服务异常状态或保留现有可重试错误处理。
- 其他兑换错误继续使用现有错误分类，不改变恢复和人工核对语义。

服务端日志和浏览器错误不得记录用户 Token、会话 Cookie 或管理员 API Key。

## 测试策略

### 配置测试

- 接受 `REDEEM_ALLOWED_GROUP_ID=24` 并解析为数字 `24`。
- 缺失配置时启动失败。
- `#24`、`0`、负数、小数、空字符串和非数字值均被拒绝。
- `.env.example` 包含该必填配置。

### 单元与接口测试

- profile schema 接受合法 `allowed_groups` 并保留该字段。
- profile schema 将缺失或非法的 `allowed_groups` 规范化为空数组并拒绝授权。
- 分组成员可以 exchange、读取 `/api/me`、prepare 和 execute。
- 非成员在 exchange 时返回 `403` 且不设置会话 Cookie。
- 已登录用户被移出分组后，`/api/me` 和 prepare 返回 `403`。
- prepare 成功后、execute 前被移出分组时，execute 返回 `403`，且不调用兑换码生成和余额扣减。
- 恢复未完成操作时权限被撤销，不能调用产生副作用的上游管理员接口。
- 用户重新加入分组后可以再次通过重验证。
- 上游认证失败、不可用和 profile 格式异常保持安全且可区分的错误响应。

### 前端与端到端测试

- 桌面、移动端和 iframe 模式均正确显示无权限文案和“重新检查”按钮。
- 无权限状态不渲染兑换表单、历史记录和恢复操作。
- 首次 exchange 被拒后，在不刷新页面的情况下重新加入分组并点击“重新检查”可以登录。
- 已有会话被撤权后进入无权限状态，重新授权后可恢复。
- 获授权用户的现有单笔、批量、复制、余额刷新和恢复流程继续通过。

## 部署与验证

生产服务器 `.env` 增加：

```env
REDEEM_ALLOWED_GROUP_ID=24
```

重新构建并启动容器后执行以下验证：

1. 健康检查正常，服务日志无配置错误或敏感信息。
2. “分销代理”分组成员可以打开工具并完成一笔最小金额测试兑换。
3. 非成员打开工具时看到无权限页面，直接调用 prepare 和 execute 均返回 `403`。
4. 测试用户被移出分组后，下一次请求立即被拒绝；重新加入后点击“重新检查”恢复。
5. 验证未授权请求没有生成兑换码，也没有扣减余额。

## 安全结论

授权来源、校验位置和副作用入口均在服务端闭合。前端隐藏仅用于用户体验，不承担安全职责。系统对配置错误和资料格式异常默认拒绝，且在每次受保护请求中读取最新 sub2api profile，因此专属分组 ID `24` 是唯一兑换权限开关。
